# Team collaboration

How to run a **shared research wiki** with your lab and external collaborators. The wiki is a git repo of Markdown files — no server required.

---

## Roles

| Who | Owns | Does |
|-----|------|------|
| **Researchers** | `raw/` | Curate PDFs, notes, and assets. Never edited by the assistant. |
| **LLM assistant** | `wiki/` (mostly) | Ingest, cross-link, synthesize, update `index.md` and `log.md`. |
| **Humans (all)** | `wiki/overview.md`, `wiki/syntheses/`, `CLAUDE.md` | Big-picture map, filed answers, schema evolution. |

The assistant **never modifies `raw/`**. Treat `raw/` as append-only from humans.

---

## What to commit

| Path | Commit? | Notes |
|------|---------|-------|
| `raw/papers/`, `raw/notes/` | **Yes** (usually) | Source of truth. See privacy below. |
| `wiki/` | **Yes** | Generated + hand-maintained pages. |
| `index.md`, `log.md` | **Yes** | Navigation and history. |
| `builder/data.py`, `builder/deepdives/` | **Yes** | Corpus definition and deep-dive source text. |
| `builder/entries/` | **Yes** | Chart entry notes (themes, abstract, one-liner). |
| `builder/templates/` | **Yes** | Default scaffolds (engine; rarely edited per vault). |
| `builder/auto_papers.json`, `auto_sources.json` | **Yes** | Auto-registry from Quick Dip / map_channel. |
| `manager/` | **Yes** | Portolan UI (optional for CLI-only teams). |
| `builder/cache/` | **No** | Regenerable PDF text; gitignored. |
| `.obsidian/workspace*.json` | **No** | Per-machine UI state; gitignored. |

## Git workflow

### Adding papers (LLM ingest path)

1. Branch from `main` (e.g. `ingest/chi2026-paper`).
2. Add PDFs/notes to `raw/`.
3. Run ingest with your LLM agent (locally).
4. Review `wiki/` changes in Obsidian or `git diff`.
5. Open a PR. A teammate spot-checks summaries and links.
6. Merge; append-only `log.md` entry documents the ingest.

### Adding papers (Portolan / builder path)

1. Dock PDF to `raw/papers/` (UI or copy manually).
2. Run **Update chart** in Portolan or `python3 builder/map_channel.py --channel my-portfolio`.
3. Edit `builder/entries/my-portfolio/<slug>.md` (themes, abstract, one-liner).
4. Fill `builder/deepdives/<slug>.md` (or wait for LLM Deep Dive in a future release).
5. Run `python3 builder/build.py` — exit code **0** means no red links.
6. Update `wiki/overview.md` if threads shifted.
7. PR with `builder/entries/`, `builder/deepdives/`, and generated `wiki/`.

Check **Status**: **On chart** (deep dive done) = ready for the team to read; **Enrich next** / **Quick dip** = still incomplete.

### Avoiding merge conflicts

- **Serialize ingests** when possible — one person/agent ingests at a time.
- `log.md` is append-only; conflicts are rare if everyone appends at the bottom.
- `index.md` is regenerated by the builder; for LLM-only vaults, the assistant rewrites it — coordinate if two ingests land at once.

## Privacy and sensitive sources

PDFs in git are **convenient but permanent**. Before committing:

- **Embargoed or in-review papers** — keep in a private repo or `raw/` submodule until public.
- **IRB-restricted data** — do not commit raw exports; ingest de-identified summaries only.
- **Collaborator drafts** — get consent before adding to a shared vault.

Options:

| Approach | When |
|----------|------|
| **Private GitHub repo** | Default for active labs |
| **Git LFS** for large PDF corpora | Many/large files |
| **Private `raw/` submodule** | Share wiki, restrict sources |
| **Public repo, no PDFs** | Open methodology; `status: no-pdf` in builder |

## External collaborators

- **Read access** — share the repo; collaborators browse in Obsidian.
- **Write access** — use PRs; don't push directly to `main` unless your team prefers trunk-based flow.
- **Fork workflow** — external partners maintain a fork; you pull selected syntheses or sources via PR.

Document your lab's choice in `wiki/overview.md` or a team README snippet.

---

## Schema changes (`CLAUDE.md`)

`CLAUDE.md` is the operating manual. Changes affect **how every future ingest behaves**:

1. Propose schema edits in a PR with a short rationale.
2. After merge, tell the team to re-read or re-ingest if conventions shifted materially.
3. Major changes get a `log.md` entry: `## [date] schema | <what changed>`.

---

## Health checks

Periodically (monthly or after a big ingest batch):

```
"Lint the wiki."
```

The assistant reports contradictions, orphans, red links, and stale claims. See `CLAUDE.md` §6 (lint).

For builder-managed vaults, also run:

```bash
python3 builder/build.py   # exit 1 = red links to fix
```
